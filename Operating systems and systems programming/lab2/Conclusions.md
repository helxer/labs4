# Висновки

В ході виконання лабораторної роботи №2 було реалізовано та досліджено CPU-інтенсивну задачу — пошук простих чисел — у послідовному та багатопроцесорному варіантах із використанням модуля `multiprocessing` у Python.

**Основні результати:**

1. **Multiprocessing розв'язує проблему GIL**. У ЛР1 для CPU-bound обробки файлів `threading` давав прискорення < 1x через Global Interpreter Lock. У цій роботі заміна потоків на окремі процеси (`multiprocessing.Pool`) забезпечила реальне паралельне виконання, оскільки кожен процес має власний інтерпретатор Python із власним GIL.

2. **Прискорення близьке до кількості ядер**. На 4 процесах отримано 2.85x, на 8 — 3.94x. Це відповідає очікуваній поведінці для CPU-bound навантаження з низькою інтенсивністю IPC.

3. **`Pool(1)` повільніший за чисто послідовне виконання** через накладні витрати на створення процесу, серіалізацію даних через pickle та збір результатів. Це підтверджує, що багатопроцесорність має сенс лише за наявності достатнього обсягу роботи на одиницю IPC-комунікації.

4. **Hyper-threading не дає лінійного приросту**. Перехід з 4 на 8 процесів збільшив прискорення лише з 2.85x до 3.94x — логічні ядра HT ділять виконавчі блоки фізичного ядра, тому для чисто-обчислювальної задачі їхня ефективність нижча.

5. **`Pool.map` зручний для embarrassingly-parallel задач** — задача природно розбивається на незалежні підзадачі, які Pool автоматично розподіляє між воркерами та збирає результати у правильному порядку.

**Практичний висновок**: для CPU-bound задач у Python використовуйте `multiprocessing`; для I/O-bound — `threading` або `asyncio`. Оптимальна кількість процесів ≈ числу фізичних ядер CPU; гіпертред-ядра дають додатковий, але вже сублінійний приріст.
