# Контрольні питання до лабораторної роботи №2

### 1. Що таке GIL і чому він заважає потокам?

**GIL (Global Interpreter Lock)** — це механізм CPython, який гарантує, що у певний момент часу лише один потік виконує байт-код Python усередині одного процесу-інтерпретатора. Це спрощує управління пам'яттю (зокрема, лічильниками посилань), але обмежує реальний паралелізм.

Для **CPU-bound** задач (як пошук простих чисел) GIL означає, що навіть на багатоядерному CPU всі потоки одного процесу будуть виконуватися по черзі на одному ядрі, постійно перемикаючись. Тому `threading` не дає прискорення для обчислювальних задач — потрібен `multiprocessing`, оскільки кожен процес має власний інтерпретатор з власним GIL.

Для **I/O-bound** задач GIL не критичний: коли потік чекає на ввід/вивід, він відпускає GIL і дозволяє іншому потоку виконуватися.

### 2. Як переглянути список процесів у Linux?

Основні команди:
- `ps aux` — повний список процесів усіх користувачів у системі з PID, %CPU, %MEM, командою.
- `ps -ef` — альтернативний формат з PPID (parent PID).
- `top` / `htop` — інтерактивний моніторинг процесів у реальному часі з сортуванням за CPU/MEM.
- `pgrep <name>` — знайти PID процесів за іменем.
- `pidof <name>` — повернути PID процесу за іменем виконуваного файлу.

Для роботи з конкретним процесом:
- `kill <PID>` — завершити процес (SIGTERM).
- `kill -9 <PID>` — примусово завершити (SIGKILL).

### 3. Як створити пул процесів у Python?

Через клас `Pool` із модуля `multiprocessing`:

```python
from multiprocessing import Pool

with Pool(processes=4) as pool:
    results = pool.map(is_prime, numbers)
```

Контекстний менеджер `with` автоматично викликає `pool.close()` та `pool.join()` після завершення. Якщо `processes` не вказано — використовується `os.cpu_count()`. Pool також надає методи `apply`, `apply_async`, `starmap`, `imap`, `imap_unordered`.

### 4. Яка різниця між `Process` та `Pool`?

- **`Process`** — створення *одного* окремого процесу під конкретну функцію. Треба самостійно керувати запуском (`start()`), очікуванням (`join()`), передачею аргументів і отриманням результату (через `Queue` чи `Pipe`). Підходить, коли процесів небагато і кожен виконує свою унікальну роботу.

- **`Pool`** — *менеджер* фіксованого набору worker-процесів. Автоматично розподіляє великий список задач між ними через `map`/`apply`, повторно використовує процеси, збирає результати у правильному порядку. Підходить для embarrassingly-parallel задач, де одна функція застосовується до багатьох даних (як у нашому пошуку простих чисел).

Коротко: `Process` — низькорівневий примітив, `Pool` — високорівневий патерн "data parallelism" над ним.

### 5. Чому для цієї задачі не використовували `threading`?

Пошук простих чисел — це **CPU-bound задача**: 100% часу витрачається на обчислення (ділення, цикли), а не на ввід/вивід. У такому режимі GIL стає вузьким місцем: усі потоки одного Python-процесу будуть конкурувати за єдиний інтерпретатор і виконуватися фактично послідовно, додаючи лише накладні витрати на перемикання контексту.

Це безпосередньо підтверджено у ЛР1 — для CPU-bound обробки файлів `threading` дав speedup < 1x. Тут `multiprocessing` дає реальний speedup ≈ 4x на 8 ядрах, оскільки кожен процес має власний інтерпретатор і працює на окремому ядрі CPU незалежно.

### 6. Як виміряти час виконання програми в Linux?

**З командного рядка:**
- `time python3 sequential_primes.py` — виводить `real` (астрономічний), `user` (час у user space) та `sys` (час у kernel) після завершення команди.
- `/usr/bin/time -v` (GNU time) — детальніша статистика: maximum resident set size, page faults, context switches тощо.
- `perf stat` — низькорівневі CPU-лічильники (instructions, cache-misses, IPC).

**Усередині Python:**
- `time.time()` — wall-clock час у секундах (використано в нашій лабі).
- `time.perf_counter()` — монотонний таймер з найвищою точністю, рекомендований для бенчмарків.
- `time.process_time()` — лише CPU-час процесу, без часу очікування на I/O.
- Модуль `timeit` — для мікро-бенчмарків з усередненням по кількох запусках.
