# Висновки

У ході виконання лабораторної роботи №3 досліджено механізми керування процесами в Linux та класичне програмне рішення проблеми взаємного виключення — алгоритм Патерсона.

**Основні результати:**

1. **Ієрархія процесів та планування**. Через `multiprocessing.Process` створено дерево процесів, де всі нащадки мають однаковий PPID = PID батька. Виводи з `time.sleep()` показали, що планувальник ОС реально чергує процеси у стані `Ready` / `Waiting`, перериваючи одного для запуску іншого.

2. **Race condition підтверджено експериментально**. Без синхронізації 2 процеси × 10 000 інкрементів дали лічильник ~10 000 замість 20 000 — втрачено ~50% оновлень. Це класичний наслідок неатомарності read-modify-write і паралельного виконання на різних ядрах.

3. **Алгоритм Патерсона усуває race condition**. Усі три тестові запуски дали рівно 20 000 — повну коректність — при накладних витратах лише ~6% часу. Алгоритм забезпечує всі три класичні умови (Mutual Exclusion, Progress, Bounded Waiting) виключно програмно, без апаратних інструкцій типу `test-and-set` чи `CAS`.

4. **Обмеження Патерсона**. Алгоритм працює лише для **двох** процесів. Узагальнення на N процесів вимагає алгоритмів типу Lamport's Bakery або Eisenberg-McGuire, які значно складніші. Крім того, на сучасних out-of-order CPU чистий Патерсон вимагає memory barriers — інакше переупорядкування записів може порушити коректність (у Python це частково приховано рівнем інтерпретатора).

5. **Практичний висновок**. У реальному коді замість Патерсона зазвичай використовують готові примітиви: `multiprocessing.Lock`, `Semaphore`, `Condition` — вони реалізовані через системні виклики (futex у Linux), що дозволяє процесу-очікувачу заснути замість busy-wait та звільнити CPU.
