# Питання для самоконтролю

### 1. Яка роль системного виклику `fork()` та як його імітує Python?

`fork()` — це системний виклик POSIX, який створює новий процес шляхом **повного дублювання** адресного простору батьківського процесу. Дочірній процес отримує копію пам'яті, відкритих файлових дескрипторів, регістрів і продовжує виконання з того самого місця, що й батько. Розрізнити їх можна за значенням, яке повертає `fork()`: батьку повертається PID нащадка, нащадку — 0.

**У Python** немає прямого еквіваленту в основній бібліотеці для прикладного коду, але:
- Модуль `os` має `os.fork()` (тільки на Unix), який є тонкою обгорткою системного виклику.
- Модуль `multiprocessing` приховує деталі: на Linux/macOS він за замовчуванням використовує **start method `fork`** (на macOS Python 3.8+ — `spawn`, який створює новий інтерпретатор «з нуля» через `exec`). Користувач просто створює `Process(target=func)` і викликає `start()` — решта обробляється бібліотекою.

### 2. Що таке контекст процесу та чому його перемикання є витратною операцією?

**Контекст процесу** — це повний стан, який ОС має зберегти/відновити, щоб продовжити процес рівно з того місця, де його було перервано:
- значення всіх регістрів CPU (включно з PC, SP, прапорцями),
- покажчик на сторінкову таблицю (CR3 на x86) — визначає адресний простір,
- стан FPU/SSE/AVX-регістрів,
- таблиця відкритих файлів, сигнальні маски,
- стан кешу TLB (Translation Lookaside Buffer).

**Витрати на перемикання:**
1. **Збереження/відновлення регістрів** — десятки наносекунд.
2. **Скидання TLB** — при зміні адресного простору TLB стає недійсним, наступні звернення до пам'яті потребують повного перегляду сторінкових таблиць (page walk).
3. **Cache pollution** — гарячі дані попереднього процесу витісняються з L1/L2-кешу даними нового.
4. **Перехід у kernel mode** — сам факт того, що перемикання робить ядро ОС, означає трамплін через перехід привілеїв.

Загалом перемикання контексту процесу займає **1–10 мікросекунд** + наступна деградація через cache misses. Перемикання потоків того ж процесу дешевше, бо не змінюється адресний простір (TLB та частково кеш зберігаються).

### 3. Чому алгоритм Патерсона важко використовувати, якщо процесів більше двох?

Класичний Патерсон побудований на **симетричній парі**: «я хочу зайти» (`flag[i]`) + «зараз твоя черга» (`turn`). Для двох процесів логіка «процес i пропускає процес 1-i» проста й симетрична.

Для **N процесів** виникають фундаментальні складнощі:
- Замість єдиної змінної `turn` потрібен механізм впорядкування всіх N процесів, які одночасно хочуть зайти в критичну секцію.
- Звичайне узагальнення вимагає каскаду парних виборів — процес проходить через N-1 «турнір» з парних Патерсонів, де на кожному рівні відсіюються «програвачі». Це дає O(N) барʼєрів і O(N²) memory.
- Альтернатива — **Lamport's Bakery Algorithm**: кожен процес бере «талон» з номером і чекає на свою чергу. Простіший концептуально, але потребує необмежених лічильників та має O(N) пам'яті та O(N) часу очікування.
- Умова **Bounded Waiting** стає набагато складнішою для доказу при N > 2.

Тому на практиці для N процесів використовують готові примітиви ОС: семафори, м'ютекси через futex, RCU, lock-free структури на CAS.

### 4. Які стани може приймати процес під час свого життєвого циклу?

Класична модель станів процесу:

- **New (Created)** — процес щойно створено (`fork()` пройшов, але exec ще не виконано або планувальник не додав у чергу).
- **Ready (Runnable)** — процес готовий до виконання та чекає у черзі планувальника, поки CPU звільниться.
- **Running** — процес зараз виконується на одному з ядер CPU.
- **Waiting (Blocked / Sleeping)** — процес чекає на подію (I/O, сигнал, звільнення м'ютекса, закінчення `sleep`). Не споживає CPU.
- **Terminated (Zombie)** — процес завершив виконання, але батько ще не зчитав код виходу через `wait()`/`waitpid()`. PID лишається зайнятим.

**Переходи:**
- `New → Ready` — планувальник прийняв процес.
- `Ready → Running` — диспетчер обрав цей процес для виконання.
- `Running → Ready` — закінчився quantum (preemption) або з'явився процес з вищим пріоритетом.
- `Running → Waiting` — процес викликав блокуючий системний виклик (`read`, `wait`, `sleep`).
- `Waiting → Ready` — подія, на яку чекав процес, відбулася.
- `Running → Terminated` — `exit()` або фатальний сигнал.

У Linux також є стани **Uninterruptible Sleep (D)** — чекання на kernel I/O, яке не можна перервати сигналом, та **Stopped (T)** — призупинено через `SIGSTOP`.
