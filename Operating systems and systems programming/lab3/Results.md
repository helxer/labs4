# Результати виконання

## Частина 1: ієрархія процесів (PID / PPID)

Скрипт `pids_demo.py` створює 3 паралельних дочірніх процеси з різними затримками `time.sleep()`, щоб продемонструвати, як планувальник ОС чергує їх виконання.

**Фрагмент виводу:**

```
=== Parent process ===  PID=23890  PPID=23887
[A] start  PID=23893  PPID=23890
[C] start  PID=23895  PPID=23890
[B] start  PID=23894  PPID=23890
[B] tick 1  PID=23894  t=00:57:3
[A] tick 1  PID=23893  t=00:57:3
[C] tick 1  PID=23895  t=00:57:3
[B] tick 2  PID=23894  t=00:57:3
[A] tick 2  PID=23893  t=00:57:3
[B] tick 3  PID=23894  t=00:57:3
[B] done   PID=23894
[C] tick 2  PID=23895  t=00:57:3
[A] tick 3  PID=23893  t=00:57:3
[A] done   PID=23893
[C] tick 3  PID=23895  t=00:57:3
[C] done   PID=23895

=== Process tree (parent=23890) ===
  child PID=23893  name=Process-1
  child PID=23894  name=Process-2
  child PID=23895  name=Process-3
```

**Спостереження:**
- У всіх трьох дочірніх процесів **PPID = 23890** — це PID батьківського процесу. Це підтверджує ієрархічне відношення «батько-нащадок», створене через `multiprocessing.Process`.
- Виводи `tick` чергуються між процесами (B → A → C → B → A → B → done → C → A → done → C → done) — планувальник ядра ОС перериває один процес у стані `Waiting` (всередині `sleep`) і запускає інший у стані `Ready`.
- Повний лог збережено у `lab_os_3/output/pids_demo.txt`.

## Частина 2: race condition (без синхронізації)

Скрипт `race_condition.py` запускає 2 процеси, кожен виконує 10 000 неатомарних інкрементів `multiprocessing.Value('i', 0, lock=False)`. Очікуване значення = 20 000.

| Запуск | Очікувано | Фактично | Втрачено оновлень |
|--------|-----------|----------|-------------------|
| 1      | 20 000    | 10 409   | 9 591             |
| 2      | 20 000    | 10 184   | 9 816             |
| 3      | 20 000    | 10 078   | 9 922             |

**Пояснення:** оператор `counter.value = counter.value + 1` у Python неатомарний — він складається з read-modify-write. Якщо планувальник перемикає процеси між read і write, обидва процеси читають однакове значення і пишуть однакове `+1`, втрачаючи один інкремент. Втрати ~50% означають, що практично кожен другий інкремент губиться.

## Частина 3: алгоритм Патерсона

Скрипт `peterson.py` використовує алгоритм Патерсона (масив `flag[2]` + змінна `turn` у shared memory) для захисту критичної секції.

| Запуск | Очікувано | Фактично | Втрачено оновлень |
|--------|-----------|----------|-------------------|
| 1      | 20 000    | **20 000** | 0               |
| 2      | 20 000    | **20 000** | 0               |
| 3      | 20 000    | **20 000** | 0               |

## Порівняльна таблиця

| Стратегія                | Результат лічильника | Час (с)  |
|--------------------------|----------------------|----------|
| Без синхронізації        | ~10 000 / 20 000     | 0.062    |
| Алгоритм Патерсона       | **20 000 / 20 000**  | 0.066    |

Накладні витрати Патерсона мінімальні (~6%), бо обидва процеси переважно встигають проходити критичну секцію без активного очікування — `flag[other]` майже завжди = `False` на момент перевірки. Зате коректність — 100%.

## Як алгоритм забезпечує три умови

**Mutual Exclusion** — якщо обидва процеси одночасно встановили `flag[i] = True`, то `turn` може мати лише одне значення (0 або 1) — тому рівно один з них «програє» в перевірці `while flag[other] and turn == other` і чекає, поки інший вийде з критичної секції та виставить `flag[i] = False`.

**Progress** — якщо критична секція вільна (`flag[other] == False`), процес негайно входить, не блокуючись.

**Bounded Waiting** — після виходу з критичної секції процес `i` встановлює `flag[i] = False`. Якщо одразу хоче знову зайти, він зробить `turn = other` — пропустить чергу другому. Тому жоден процес не може зайти у критичну секцію двічі підряд, поки інший чекає.
